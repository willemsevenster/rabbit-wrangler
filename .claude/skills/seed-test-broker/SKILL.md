Populate a RabbitMQ broker with realistic-looking sample data so Rabbit Wrangler
has something to peek, move, purge, and navigate. The rig is a single Node script
â€” `scripts/seed-broker.mjs` (pure `amqplib`, already a project dependency) â€” driven

All paths are relative to the project root (`rabbit-wrangler/`). Verified on
Windows against a local broker.

## Prerequisites

- A reachable RabbitMQ broker with the **management plugin** (for guest/guest +
  the HTTP API). Locally:
  ```sh
  docker run -d --name rabbit -p 5672:5672 -p 15672:15672 rabbitmq:management
  ```
- `pnpm install` (brings in `amqplib`, which the script imports).

## Run (agent path)

One-shot seeds â€” pick the volume with `--messages` (a few, or many):

```sh
node scripts/seed-broker.mjs --messages 8      # few
node scripts/seed-broker.mjs --messages 40     # default-ish
node scripts/seed-broker.mjs --messages 1000   # many
```

`pnpm seed` is an alias (forward flags after `--`):

```sh
pnpm seed -- --messages 40
```

Live traffic (for testing background peeking + unread badges) â€” publishes
continuously until Ctrl-C:

```sh
node scripts/seed-broker.mjs --rate 20
```

Tear everything down:

```sh
node scripts/seed-broker.mjs --clean
```

Other flags: `--url <amqp-url>` (or `$RABBIT_URL`; default
`amqp://guest:guest@localhost:5672`), `--prefix <name>` (default `rw.demo`),
`--durable` (persist queues/messages across broker restarts), `--help`.

## What it creates

A topic exchange `rw.demo.events` with four bound queues â€”
`rw.demo.orders` (`order.#`), `rw.demo.payments` (`payment.#`),
`rw.demo.notifications` (`notify.#`), and `rw.demo.audit` (`#`, catch-all) â€” plus
two dead-letter queues `rw.demo.orders.dlq` / `rw.demo.payments.dlq`.

Messages are **randomly varied**: JSON (orders / payments / notifications / user
events), plain-text log lines, and occasional binary blobs; mixed routing keys;
mixed properties (â‰ˆ70% carry a `messageId`, the rest exercise the body-hash
fingerprint; some `correlationId`, `timestamp`, `appId`, custom `x-*` headers).

## Gotchas

- **DLQ x-death is generated by real dead-lettering, not faked.** RabbitMQ
  **strips a client-supplied `x-death` header** (resets it to `[]`). So the script
  populates the `.dlq` queues by publishing to a transient feeder queue with
  `x-message-ttl: 0` + a dead-letter route, letting the broker add an authentic
  `x-death` (reason `expired`). The feeders (`*.dlq.src`) are deleted after the
  broker finishes dead-lettering.
- **Transient by default.** Queues/messages are non-durable unless `--durable`, so
  a broker restart clears them. Use `--clean` for an explicit teardown.
- **Live mode floods fast.** `--rate` runs until interrupted; stop it with Ctrl-C.
  Don't background it from a harness that can't deliver SIGINT, or it keeps
  publishing. `--clean` afterwards resets the counts.
- **Stats lag.** The management API samples queue stats every ~5s, so counts can
  read `undefined`/stale for a moment right after seeding.
